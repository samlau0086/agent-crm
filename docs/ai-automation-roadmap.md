# AI 自动化落地 Roadmap

> 文档状态：Draft v1  
> 适用项目：AI Agent CRM  
> 建议周期：12–16 周（按 2–4 名工程师估算）  
> 核心原则：先可靠、再智能、最后开放平台能力

> 当前上线状态：通用工作流自动化模块默认下线。`WORKFLOW_AUTOMATION_ENABLED=false` 时，导航、页面、API、事件入队、定时扫描和等待恢复均关闭；数据库、历史运行与实现代码保留，供后续按本 Roadmap 重构和灰度恢复。

## 1. Roadmap 目标

本 Roadmap 用于把当前偏通用、但运行稳定性不足的工作流模块，逐步收敛并建设为一个可信赖的 AI 客户跟进系统。

目标 MVP 不以“支持更多节点”或“创建更多工作流”为成功标准，而以以下业务结果为目标：

- 重要客户邮件不会漏处理。
- 商机长时间无人跟进时能够及时发现。
- 客户承诺、日期和下一步行动能够被提取并持续追踪。
- AI 能够生成有依据的下一步建议、邮件草稿和内部任务。
- 任何对外发送或重要 CRM 修改都受到明确策略和审批控制。
- 每一次自动化运行都可解释、可恢复、可重试、可审计。

最终产品定位：

> AI 客户跟进管家：理解客户沟通、识别商机风险、准备下一步动作，并持续追踪结果。

## 2. 产品边界

### 2.1 MVP 首批支持的触发器

- 收到客户邮件。
- CRM 记录创建。
- CRM 记录更新。
- 每日定时扫描。
- 用户手动运行。

### 2.2 MVP 首批支持的判断能力

- CRM 字段条件。
- 距离上次联系的时间。
- 是否已收到客户回复。
- 邮件类型、意图和紧急程度。
- AI 置信度。
- 是否存在未完成的同类任务或建议。

### 2.3 MVP 首批支持的动作

- 创建邮件草稿。
- 创建任务或提醒。
- 添加内部标签或备注。
- 发送内部通知。
- 生成 CRM 字段修改建议。
- 经审批后发送邮件或修改 CRM 关键字段。

### 2.4 MVP 暂不支持的能力

- 新增更多通用流程节点。
- 无界循环和复杂嵌套循环。
- 任意脚本执行。
- 任意 HTTP 请求。
- AI 自由选择全部工具。
- 自动批量发送邮件。
- 无审批修改金额、阶段或负责人。
- 尚未完整实现的 weekly、interval 等触发器。
- 运行时动态修改工作流自身。

## 3. 目标用户体验

自动化模块应形成三个层级，而不是默认要求所有用户理解流程画布。

### 3.1 AI 待处理箱

销售人员每天首先看到：

- 待回复的重要客户邮件。
- AI 准备好的邮件草稿。
- 高风险或长时间未跟进的商机。
- 即将逾期的客户承诺。
- 等待批准的 CRM 修改。
- 执行失败且需要人工处理的自动化。

每一条建议必须回答：

- 为什么产生这条建议。
- 使用了哪些数据和来源。
- 系统准备执行什么动作。
- 动作属于什么风险级别。
- 是否会联系外部人员或修改业务数据。

### 3.2 自动化方案

用户可以通过模板或自然语言表达目标，例如：

> 客户询价后准备回复，如果三天没有回复就提醒负责人。

系统将其转换为受支持、可验证的结构化方案，并在启用前展示触发范围、审批要求和历史模拟结果。

### 3.3 高级流程编辑器

仅向管理员或高级用户开放，用于调整受支持的条件、等待时间、审批策略和动作，不作为目标 MVP 的主要产品入口。

## 4. 总体交付路径

| 阶段 | 建议周期 | 主要结果 | 上线门槛 |
| --- | --- | --- | --- |
| 准备阶段（Phase 0） | 1 周 | 能力边界和风险规则明确 | 未完成能力不再对外承诺 |
| 1. 可靠执行内核 | 3–4 周 | 事件不丢、动作不重、流程可恢复 | 核心稳定性验收通过 |
| 2. 跟进管家 MVP | 3–4 周 | 邮件、失联商机、客户承诺形成闭环 | 内测业务指标达标 |
| 3. 信任型体验 | 2–3 周 | 建议箱、模拟、运行时间线 | 用户能够理解和修复运行 |
| 4. 模板化扩展 | 2 周 | 可复制的行业自动化模板 | 模板均通过模拟与风险审查 |
| 5. 有限开放平台 | 持续建设 | 高级流程编辑与连接器 | 可靠语义和治理能力成熟 |

## 5. 准备阶段（Phase 0）：范围收敛与基线建立

### 5.1 工作项

- 形成触发器、条件、动作和节点的正式支持矩阵。
- 隐藏或标记尚未具备完整运行语义的节点。
- 盘点现有工作流，识别需要迁移、停用或降级的流程。
- 定义动作风险分级和默认审批策略。
- 记录当前运行成功率、失败率、重复率和卡住运行数量。
- 建立 feature flag，确保新内核可以灰度启用和快速关闭。

### 5.2 风险分级

#### 低风险：允许自动执行

- 摘要、分类和结构化提取。
- 添加内部标签。
- 创建内部提醒。
- 创建邮件草稿。
- 发送内部通知。

#### 中风险：默认生成建议或草稿

- 修改普通客户字段。
- 创建对外回复内容。
- 修改预计跟进日期。
- 分配负责人。

#### 高风险：必须审批

- 真正发送邮件。
- 修改商机阶段、金额或预计成交日期。
- 删除数据。
- 批量操作。
- 调用对外 webhook。
- AI 自动调用修改型工具。

### 5.3 阶段验收

- 支持矩阵经过产品和工程共同确认。
- 所有未完成能力均被隐藏或明确标为实验性。
- 当前稳定性指标已经形成可重复生成的基线报告。
- 每种动作都有明确的风险等级和审批策略。

## 6. 阶段 1：可靠执行内核

此阶段是所有后续 AI 能力的前置条件。

### 6.1 持久化事件账本

业务操作产生事件时，应在数据库事务内同时写入业务数据和 `AutomationEvent`，再由后台 worker 异步消费。

建议字段：

```text
AutomationEvent
- id
- workspaceId
- eventType
- sourceType
- sourceId
- payload
- occurredAt
- availableAt
- status
- attempts
- claimedBy
- claimedAt
- processedAt
- lastErrorCode
- lastErrorMessage
- idempotencyKey
```

事件状态建议：

```text
pending → processing → completed
                   ↘ retrying → processing
                   ↘ failed
```

要求：

- Redis 不可用时事件仍然保存在数据库中。
- worker 重启后能够继续消费。
- 同一业务事件只能产生一次有效运行。
- 支持按 workspace、事件类型和时间范围查询。

### 6.2 工作流版本化

启用中的流程必须引用不可变版本。用户编辑时创建新版本，历史运行继续引用原版本。

建议新增 `WorkflowVersion`：

```text
- id
- workflowId
- version
- definition
- definitionHash
- publishedAt
- publishedById
- status
```

运行记录保存 `workflowVersionId`，避免流程更新后历史运行无法解释。

### 6.3 节点级运行状态

建议新增或强化 `WorkflowStepRun`：

```text
- id
- workspaceId
- runId
- nodeId
- nodeType
- attempt
- status
- inputSnapshot
- output
- startedAt
- completedAt
- nextAttemptAt
- errorCode
- errorMessage
- idempotencyKey
```

工作流运行状态：

```text
queued
→ running
→ waiting / approval_required / retrying
→ running
→ completed / failed / cancelled
```

要求：

- 每个节点执行前先持久化运行状态。
- 每个副作用完成后先保存结果，再进入下一节点。
- 进程异常后可以从最后一个安全检查点继续。
- 不允许运行永久停留在 `running`。

### 6.4 原子抢占和并发控制

事件、恢复任务、审批任务必须使用原子状态转换：

```text
pending → processing
```

只有成功完成状态转换的 worker 才能执行任务。

需要覆盖：

- 多 worker 同时扫描同一事件。
- 多 worker 同时恢复等待任务。
- 用户重复点击审批。
- API 请求超时后客户端重试。
- worker 执行过程中崩溃。

### 6.5 节点级幂等

每个产生副作用的节点使用稳定幂等键：

```text
workspaceId + runId + nodeId + operationPurpose
```

示例：

- 邮件草稿或发送：`workflow:{runId}:{nodeId}`。
- 任务创建：`workflow:{runId}:{nodeId}`。
- 通知：`workflow:{runId}:{nodeId}:{channel}`。
- CRM 更新：目标记录版本加 patch hash。

对于外部系统，还应把该幂等键传递给对方，或在本地保存外部请求与结果映射。

### 6.6 审批作为可恢复节点

正确执行路径：

```text
到达高风险节点
→ 固化待执行参数与输入快照
→ 运行进入 approval_required
→ 用户批准或拒绝
→ 原子领取审批
→ 执行当前节点
→ 保存节点结果
→ 从下一节点继续
→ 更新运行最终状态
```

审批记录至少保存：

- 动作摘要和风险说明。
- 业务数据来源。
- 待执行参数快照。
- 工作流版本。
- 申请人和审批人。
- 审批意见。
- 过期时间。
- 最终执行结果。

拒绝后必须进入明确的拒绝分支、结束流程或退回修改，不能留下无法解释的运行状态。

### 6.7 错误分类和重试策略

建议统一错误代码：

- `CONFIGURATION_ERROR`
- `VALIDATION_ERROR`
- `AUTHENTICATION_ERROR`
- `PERMISSION_DENIED`
- `PROVIDER_UNAVAILABLE`
- `RATE_LIMITED`
- `RECORD_NOT_FOUND`
- `CONFLICT`
- `TIMEOUT`
- `INTERNAL_ERROR`

重试原则：

- 配置、权限和校验错误不自动重试。
- 限流、网络超时和服务暂时不可用采用指数退避。
- 达到最大次数后进入 dead letter 状态。
- 管理员可在修复配置后重试单个节点。

### 6.8 运行看门狗

- 周期扫描超时的 `processing` 和 `running` 记录。
- 对可恢复运行释放 claim 并重新调度。
- 对状态不一致的运行标记异常并通知管理员。
- 输出 pending、running、retrying、failed 和 dead letter 数量。

### 6.9 阶段 1 测试矩阵

- 同一事件并发提交 10 次，只产生一次副作用。
- 动作执行成功但结果写入前进程中断，恢复后不重复动作。
- 两个 worker 同时领取恢复任务，只有一个成功。
- 审批按钮重复点击，不产生重复动作。
- Redis 暂时不可用，恢复后事件仍能处理。
- AI provider、SMTP、数据库分别超时时状态正确。
- 流程版本更新不改变历史运行解释。
- 等待节点跨进程重启后能够按时继续。

### 6.10 阶段 1 上线门槛

- 核心自动化运行成功率不低于 99.5%。
- 非预期重复外部动作数量为 0。
- 已提交但永久丢失事件数量为 0。
- 审批后恢复成功率达到 99.9%。
- 所有失败运行都有错误代码、错误阶段和恢复建议。
- 不存在超过规定超时时间仍无解释地停留在 `running` 的记录。

## 7. 阶段 2：AI 客户跟进管家 MVP

### 7.1 场景 A：新客户邮件处理

处理步骤：

1. 接收并持久化邮件事件。
2. 关联联系人、公司和商机。
3. 识别邮件类型、客户意图、情绪和紧急程度。
4. 提取产品、数量、预算、交期、承诺和下一日期。
5. 生成带来源的简短摘要。
6. 生成推荐动作和原因。
7. 创建回复草稿。
8. 必要时创建任务或提醒。
9. 对 CRM 修改生成待批准建议。

第一版邮件分类建议：

- 询价。
- 产品或技术问题。
- 交期问题。
- 报价接受或拒绝。
- 付款问题。
- 投诉或升级。
- 普通沟通。
- 无需处理。

### 7.2 场景 B：失联商机监控

每日先使用确定性规则筛选候选记录，再交给 AI 分析，避免模型扫描全部数据。

候选条件可包括：

- 超过阈值天数无外部联系。
- 存在未回复的客户邮件。
- 已发送报价但长期没有进展。
- 存在即将到期或已逾期承诺。
- 没有未完成的同类跟进任务。

AI 输出：

- 风险等级和原因。
- 推荐跟进时间。
- 推荐沟通渠道。
- 回复或跟进草稿。
- 是否需要提醒负责人或主管。

### 7.3 场景 C：客户承诺追踪

从邮件和活动中提取：

- 承诺人。
- 承诺内容。
- 承诺日期。
- 关联客户和商机。
- 来源邮件或活动。
- 当前完成状态。

建议新增 `CustomerCommitment`：

```text
- id
- workspaceId
- recordId
- sourceType
- sourceId
- ownerType
- content
- dueAt
- status
- confidence
- evidence
- createdByAutomationRunId
- completedAt
```

到期前创建提醒，到期后检查是否完成；无法判断时生成建议，不自动断言客户违约。

### 7.4 AI 结构化输出

AI 输出必须通过 JSON Schema 或等价的运行时 Schema 校验。示例：

```json
{
  "classification": "quotation_request",
  "confidence": 0.93,
  "summary": "客户希望获取 500 件产品的价格和四周内交付方案。",
  "extractedFacts": [
    {
      "field": "quantity",
      "value": 500,
      "sourceMessageId": "message-id"
    }
  ],
  "recommendedActions": [
    {
      "type": "create_email_draft",
      "reason": "客户明确要求报价",
      "risk": "low"
    }
  ],
  "needsHumanReview": true
}
```

服务端必须执行：

- Schema 校验。
- 字段和动作白名单校验。
- 用户权限校验。
- 风险和审批策略校验。
- 来源有效性校验。
- 置信度阈值判断。
- 输入输出长度和成本预算控制。

### 7.5 置信度策略

| 置信度 | 默认策略 |
| --- | --- |
| 高 | 可自动分类、摘要、打标签和创建内部提醒 |
| 中 | 生成建议或草稿，不自动修改关键数据 |
| 低 | 提示信息不足，交由用户确认 |

模型只能给出置信度和建议，不能自行绕过权限或审批策略。

### 7.6 建议与执行分离

建议增加 `AutomationSuggestion`：

```text
- id
- workspaceId
- runId
- suggestionType
- title
- reason
- evidence
- confidence
- riskLevel
- proposedAction
- status
- reviewedById
- reviewedAt
- executedStepRunId
```

标准路径：

```text
AI 产生建议
→ 策略引擎校验
→ 自动批准或用户确认
→ 转换为确定性执行命令
→ Action Executor 执行
```

### 7.7 阶段 2 上线门槛

- 邮件分类和事实提取完成离线评测集。
- 提取事实必须保存可追溯来源。
- AI 输出校验失败不会产生任何副作用。
- AI provider 不可用时能够降级并清楚显示原因。
- 邮件草稿、任务和建议均具备稳定幂等键。
- 高风险动作无法绕过审批。

## 8. 阶段 3：信任型产品体验

### 8.1 AI 待处理箱

建议按以下顺序展示：

1. 待审批动作。
2. 高风险商机。
3. 待回复的重要邮件。
4. 即将逾期的承诺。
5. 自动化失败。
6. AI 无法确定、需要用户判断的记录。

支持批准、修改后批准、拒绝、忽略、稍后处理和查看来源。

### 8.2 自然语言创建自动化

采用“编译”而非自由生成执行逻辑的方式：

```text
自然语言目标
→ 结构化意图
→ 支持能力匹配
→ 受约束流程定义
→ 静态校验
→ 历史数据模拟
→ 用户确认
→ 发布不可变版本
```

如果需求超出支持范围，应明确指出缺少的能力，不能静默生成一个行为不同的 fallback 流程。

### 8.3 历史数据模拟

启用前支持使用过去 7 天或 30 天数据进行 dry-run：

- 预计触发次数。
- 命中的客户或商机。
- 预计创建的任务、草稿和建议数量。
- 需要审批的次数。
- 被条件排除的记录及原因。
- 预计 AI 调用量和成本。

模拟模式不得产生真实业务副作用。

### 8.4 运行时间线

示例：

```text
10:02:11 收到客户邮件
10:02:12 关联到联系人 Alice
10:02:14 判断为询价，置信度 93%
10:02:15 提取数量 500，来源为客户邮件第 2 段
10:02:18 创建回复草稿
10:02:19 建议修改商机阶段，等待审批
```

每个节点支持查看输入摘要、输出、依据、耗时、重试次数和错误恢复建议。敏感正文和密钥不得写入普通审计日志。

### 8.5 AI fallback 可见性

必须明确展示本次结果属于：

- AI provider 正常生成。
- 确定性规则生成。
- AI 调用失败后的 fallback。
- 因策略或预算限制而跳过。

禁止用静默 fallback 让用户误以为 AI 已正确理解需求。

## 9. 阶段 4：模板化扩展

优先模板：

1. 新询价自动处理。
2. 客户邮件回复草稿。
3. 商机长期未跟进提醒。
4. 报价发出后的跟进。
5. 客户承诺日期提醒。
6. 重要客户邮件升级。
7. 投诉邮件通知主管。
8. 成交后的内部交接任务。
9. 付款到期提醒。
10. 客户生日或周年提醒。

每个模板必须包含：

- 适用场景。
- 默认触发条件。
- 必需数据和连接器。
- 默认动作及风险说明。
- 审批策略。
- 重复预防策略。
- 历史模拟入口。
- 推荐业务指标。

AI 可以帮助配置模板，但不能发明系统未支持的触发器、节点或动作。

## 10. 阶段 5：有限开放自动化平台

### 10.1 可以逐步开放

- 条件分支。
- 有上限的等待和延迟。
- 回复检查。
- 模板化 AI 分析节点。
- 创建草稿、任务和提醒。
- 内部通知。
- 审批节点。
- 经过认证的连接器动作。

### 10.2 继续限制

- 无界循环。
- 任意脚本和任意网络请求。
- AI 自由调用全部工具。
- 无限制批量执行。
- 绕过权限、策略和审批。
- 动态修改运行中的工作流定义。

### 10.3 开放新动作的准入标准

- 有明确输入输出 Schema。
- 有稳定幂等策略。
- 有权限和风险等级。
- 有超时、重试和错误分类。
- 能在模拟模式下执行。
- 能输出用户可理解的运行记录。
- 对外部副作用有重复预防或补偿方案。

## 11. 推荐技术架构

| 层级 | 职责 |
| --- | --- |
| Event Ledger | 持久化业务事件并保证不丢失 |
| Orchestrator | 创建、暂停、恢复、取消和结束运行 |
| Policy Engine | 权限、风险、审批、频率和预算控制 |
| AI Decision Service | 分类、提取、总结和推荐动作 |
| Action Executor | 邮件、任务、CRM 更新和通知 |
| Observability | 指标、日志、运行时间线和告警 |

关键原则：

- AI 负责理解和建议，不直接拥有业务权限。
- Policy Engine 是动作能否执行的唯一判断入口。
- Action Executor 只接收校验过的结构化命令。
- 所有外部副作用都必须幂等和可审计。
- 工作流定义、AI 决策和实际执行结果分别保存。

## 12. 现有实现迁移策略

### 12.1 双轨运行

- 为新内核增加 feature flag。
- 旧流程保持只读或限制编辑。
- 新建流程默认使用新内核。
- 选择少量内部 workspace 灰度。

### 12.2 Shadow 模式

新内核读取真实事件并生成运行结果，但不执行副作用。比较：

- 新旧触发数量。
- 条件判断结果。
- 动作计划。
- 运行耗时。
- 错误率。

### 12.3 迁移顺序

1. 只包含低风险动作的简单流程。
2. 包含等待节点的流程。
3. 包含审批节点的流程。
4. 邮件发送和 CRM 修改流程。
5. 无法映射的复杂流程保持停用并要求人工重建。

### 12.4 回滚原则

- 新内核按 workspace 或 workflow 单独关闭。
- 关闭后停止领取新事件，不中断已产生外部副作用的节点。
- 等待和审批中的运行保留快照，可在修复后继续。
- 不采用破坏性数据库回滚删除运行历史。

## 13. 可观测性与告警

### 13.1 稳定性指标

- 事件进入运行的成功率。
- 工作流和节点完成率。
- 重复副作用次数。
- 永久丢失事件数。
- 平均重试次数。
- 卡住运行数量。
- dead letter 数量。
- 审批后恢复成功率。
- P50、P95 执行延迟。

### 13.2 产品价值指标

- AI 建议采用率。
- 邮件草稿采用率。
- 草稿平均修改比例。
- 客户首次响应时间。
- 逾期未跟进商机下降比例。
- 承诺事项按时完成率。
- 每位销售每周节省时间。
- 每个有效动作需要的审批次数。

### 13.3 AI 质量指标

- 分类准确率。
- 事实提取准确率。
- 无来源事实比例。
- 低置信度正确拦截率。
- 用户拒绝建议的原因分布。
- provider fallback 比例。
- Schema 校验失败比例。
- 单次有效建议的 token 和费用。

### 13.4 告警建议

- pending 事件积压超过阈值。
- running 运行超过最大允许时间。
- 同一动作出现幂等冲突。
- 审批恢复失败。
- provider fallback 比例短时间显著升高。
- 邮件发送或 CRM 更新动作连续失败。
- dead letter 数量增加。

## 14. 测试与发布策略

### 14.1 测试层级

- 单元测试：条件、状态转换、策略和 Schema。
- 属性测试：幂等键、版本和状态机不变量。
- 集成测试：数据库、Redis、邮件 provider 和 worker。
- 故障注入：超时、重启、重复消息和部分成功。
- E2E：建议、审批、等待、恢复和运行时间线。
- 离线 AI 评测：分类、提取、依据和建议质量。

### 14.2 发布顺序

1. 本地和 CI 故障注入测试。
2. 测试 workspace shadow 模式。
3. 内部用户只读建议模式。
4. 自动执行低风险动作。
5. 开放高风险动作审批。
6. 小比例生产 workspace 灰度。
7. 达到稳定性门槛后逐步扩大。

### 14.3 发布阻断条件

- 出现事件永久丢失。
- 出现重复外部发送或重复关键 CRM 修改。
- 审批能够被绕过。
- AI 输出未经过 Schema 和策略校验即执行。
- 运行失败无法定位具体节点。
- 无法按 workspace 快速关闭新能力。

## 15. 优先级 Backlog

### P0：阻塞产品上线

- 持久化事件账本。
- 原子任务抢占。
- 工作流版本化。
- 节点级运行记录和幂等。
- 审批后恢复后续流程。
- 统一错误分类、重试和 dead letter。
- 卡住运行看门狗。
- 隐藏未真正支持的节点和触发器。

### P1：形成核心业务价值

- 新邮件分类、提取和回复草稿。
- 邮件与 CRM 记录关联质量提升。
- 失联商机扫描。
- AI 待处理箱。
- 历史模拟。
- 运行时间线。
- 建议与执行分离。

### P2：提高复用和智能程度

- 客户承诺提取与追踪。
- 自然语言生成受约束方案。
- 自动化模板库。
- 用户反馈和离线评测闭环。
- 成本、频率和 workspace 预算治理。

### P3：平台化扩展

- 高级流程编辑器。
- 经过认证的第三方连接器。
- 更多业务对象和行业模板。
- 管理员级策略配置。

## 16. 建议里程碑

### M1：可靠内核

定义：任何事件都不丢、任何副作用都不重复、任何失败都可定位和重试。

### M2：跟进管家内测

定义：新客户邮件能够稳定产生摘要、结构化事实、回复草稿和跟进任务。

### M3：团队可用

定义：加入失联商机、客户承诺、审批中心和运行时间线，销售团队可以日常使用。

### M4：自助配置

定义：用户可以从模板或自然语言创建方案，并用历史数据模拟后发布。

### M5：自动化平台

定义：在可靠执行语义、权限和治理基础上，有限开放高级编排和连接器。

## 17. 第一轮建议拆票

第一轮开发建议优先拆成以下 Epic：

1. `Automation Event Ledger`：事件表、事务写入、扫描和消费。
2. `Workflow State Machine`：运行状态、节点状态和合法转换。
3. `Atomic Claiming`：事件、恢复任务和审批的原子领取。
4. `Step Idempotency`：邮件、任务、通知和 CRM 更新的幂等。
5. `Approval Continuation`：审批后恢复当前节点和后续流程。
6. `Workflow Versioning`：发布版本和历史运行绑定。
7. `Error & Retry Policy`：错误码、重试、dead letter 和人工重试。
8. `Automation Observability`：指标、卡住运行扫描和告警。
9. `Inbox AI Decision Schema`：邮件分类、事实提取和证据结构。
10. `Automation Suggestions`：建议记录、审核和确定性执行。

每个 Epic 都必须包含数据库迁移、服务实现、权限检查、审计日志、单元测试、集成测试、故障测试和上线/回滚说明。

## 18. 决策原则

后续所有自动化需求使用以下顺序评估：

1. 是否解决高频、明确、可衡量的客户跟进问题。
2. 是否可以用确定性规则先缩小候选范围。
3. AI 输出是否能够结构化并提供来源。
4. 动作是否有明确风险等级和权限。
5. 是否具备稳定幂等、失败恢复和审计方案。
6. 是否能在历史数据上模拟。
7. 用户是否能理解为什么触发和如何修复。

如果以上任一关键问题没有答案，该能力不应直接进入自动执行阶段。

## 19. Roadmap 完成定义

完成本 Roadmap 不等于拥有最多的节点，而是达到以下状态：

- 用户首先信任系统每天推荐的少量高价值动作。
- 低风险动作能够稳定自动完成。
- 高风险动作始终受到清晰审批和权限控制。
- AI 的每个关键事实和建议都有可追溯依据。
- 系统故障不会导致事件丢失或重复联系客户。
- 自动化结果能够持续改善响应时间、跟进完整度和销售效率。

在达到这些标准之后，再扩大通用流程画布和第三方连接器，平台能力才会建立在可信赖的基础上。
